# 帳票入出力仕様書：JFP1080 事業別予算査定一覧表（当初）

**作成日:** 2026-06-23
**対象帳票:** JFP1080 事業別予算査定一覧表（当初）
**code_id:** `JFP1080_事業別予算査定一覧表(当初)`
**report_type:** `project_budget_assessment`
**実装（実行ファイル）:** `server/app/services/data_prep_reports/JFP1080.py`
**入力スキーマ:** `server/app/schemas/print_external_payload/project_assessment.py`

> 本書は実装 `JFP1080.py`（変換プログラム）を正として、帳票の **入力 → 処理 → 出力** を文書化したものである。記述と実装が食い違う場合は実装を正とし、本書を改訂すること。

---

## 1. 概要

事業ごとに予算査定額の推移（見積要求 → 1〜4次査定 → 最終査定）と前年度額・比較金額を一覧する集計表。

- **階層構造:** 事業 ＞ 施策 ＞ 施策内訳 の 3 階層（ツリー）
- **列構成:** 11 列固定（`header_col1` 〜 `header_col11`）
- **明細:** ツリーを再帰展開し、各ノードを 1 行として出力。子ノードは `items` にネスト（n 階層対応）
- **予算種別:** 当初（INITIAL）固定。補正版は別帳票 JFP2050（`revised_assessment_by_project`）

---

## 2. データフロー（入力経路）

```
[予算編成バックエンド：DBクローズ]
    │ {report_type, context, data}        ← DB-close format
    ▼
[print_external.py：外部APIルータ]
    │ ProjectAssessmentInitialPrintRequest で受信・検証
    ▼
[data_prep_report_transformer.py：transform_report()]
    │ _REGISTRY["project_budget_assessment"] で振り分け
    │   ├─ resolve_code_id() → "JFP1080_事業別予算査定一覧表(当初)"
    │   └─ transform()       → {"formData": {...}}
    ▼
[data_binding.py：formData → SVGテンプレートへバインド]
    ▼
[PDF 出力]
```

- 受信形式の判定は `is_report_format()`（`report_type`/`context`/`data` の 3 キーを持つか）。
- レジストリ登録（`data_prep_report_transformer.py`）:

  | 項目 | 値 |
  |---|---|
  | report_type | `project_budget_assessment` |
  | config | `JFP1080.PROJECT_BUDGET_ASSESSMENT` |
  | resolve_code_id | `JFP1080.resolve_code_id` |
  | transform | `JFP1080.transform` |

- **絶対ルール:** 1 code_id = 1 config = 1 専用 transform 関数（共用・分岐禁止）。

## 3. 入力仕様

### 3-1. リクエスト全体（`ProjectAssessmentInitialPrintRequest`）

| フィールド | 型 | 必須 | 説明 |
|---|---|---|---|
| `report_type` | `"project_budget_assessment"` | ○ | 固定値 |
| `context` | `PrintContextBase` | ○ | 後述 |
| `data` | `ProjectAssessmentInitialData` | ○ | `tree` を保持 |
| `sync` | bool（既定 true） | - | 同期出力フラグ |

### 3-2. context（`PrintContextBase`）

| フィールド | 型 | 必須 | 説明 |
|---|---|---|---|
| `frame_id` | int | ○ | 予算年度フレーム ID |
| `fiscal_year` | int | ○ | 会計年度（西暦） |

> JFP1080 の context は `PrintContextBase`（`plan_type` を持たない）。このため `resolve_code_id` は既定 `INITIAL` に解決され、常に `JFP1080_事業別予算査定一覧表(当初)` となる。

### 3-3. 明細データ（`data.tree[]` = `ProjectAssessmentInitialNode`）

`tree` は事業（最上位）ノードのルート配列。各ノードは `children` で下位階層を再帰的に保持する。

| フィールド | 型 | 既定 | 説明 | transform での扱い |
|---|---|---|---|---|
| `code` | str? | None | コード（project.code） | ラベル先頭に連結 |
| `name` | str? | None | 名称（project.name） | ラベルに連結 |
| `level` | int | （必須） | 階層レベル | **未使用**（後述） |
| `requested_amount` | int | 0 | 見積要求額 | col2 |
| `assess_1` | int | 0 | 1次査定 | col3 |
| `assess_2` | int | 0 | 2次査定 | col4 |
| `assess_3` | int | 0 | 3次査定 | col5 |
| `assess_4` | int | 0 | 4次査定 | col6 |
| `assess_final` | int | 0 | 最終査定額 | col7 |
| `prev_year` | int | 0 | 前年度予算額 | col8 |
| `children` | Node[] | [] | 下位ノード | 再帰展開 → `items` |

> **注意（スキーマと実装の差異）:** スキーマ上 `level` は必須だが、`transform` は階層を **再帰の深さ（depth）** から決定しており `level` を参照しない。インデント段数は入力 `level` ではなく実際のツリー構造で決まる。

---

## 4. 処理仕様（`transform`）

入力 `data.tree[]` を再帰展開し、各ノードを 1 行（dict）に変換する。

### 4-1. ラベル生成（col1）

```
label = "　"×depth + (code + " " + name).strip()
```

- インデントは全角空白 `　` を depth 段ぶん前置（ルート=0 段）。
- `code` / `name` が None の場合は空文字として連結し前後を trim。

### 4-2. 査定額の補完（カスケード）

未設定（falsy）の査定列は直前段の値を引き継ぐ：

| 列 | 値 | 補完ロジック |
|---|---|---|
| col2 見積要求額 | `req` = `requested_amount`（既定0） | - |
| col3 1次査定 | `a1` = `assess_1` or `req` | 未設定なら要求額 |
| col4 2次査定 | `a2` = `assess_2` or `a1` | 未設定なら前段 |
| col5 3次査定 | `a3` = `assess_3` or `a2` | 〃 |
| col6 4次査定 | `a4` = `assess_4` or `a3` | 〃 |
| col7 最終査定 | `af` = `assess_final` or `a4` | 〃 |
| col8 前年度額 | `prev` = `prev_year`（既定0） | - |

> 補完判定は Python の真偽値（`or`）。**0 や None は「未設定」扱い**となり前段値で上書きされる点に注意。

### 4-3. 比較金額（col9）と col10/col11

| 列 | 内容 | 実装 |
|---|---|---|
| col9 | 比較金額（最終査定 − 前年度） | `af - prev` |
| col10 | 比較率（見出し定義上） | **空文字固定（未算出）** |
| col11 | 予備（見出し定義上） | **空文字固定** |

> ヘッダ定義（docstring）では col9〜11 を「比較金額/率/予備」とするが、**現実装で値が入るのは col9 のみ**。比較率（col10）は算出していない。仕様確定時に要否を要確認。

### 4-4. 数値整形（`_fmt_int`）

| 入力 | 出力 |
|---|---|
| None / 空文字 | `""` |
| 0 | `"0"` |
| 正数 | 3 桁カンマ区切り（例 `1,234,567`） |
| 負数 | `△` + 絶対値カンマ区切り（例 `△1,234`） |
| 整数変換不可 | `str(値)` をそのまま |

### 4-5. 子ノードのネスト

`children` を持つノードは `row["items"] = [_build(子, depth+1) ...]` として再帰的に格納（n 階層対応）。

## 5. 出力仕様（`formData`）

`transform` の戻り値は `{"formData": {...}}`。

```jsonc
{
  "formData": {
    "title": "事業別予算査定一覧表(当初)",   // config.title 固定
    "subtitle": "",                          // 固定で空
    "fiscal_year": "2026",                   // str(context.fiscal_year)
    "report_date": "",                       // 固定で空
    "items": [
      {
        "header_col1": "　001 ○○事業",      // ラベル（インデント込み）
        "header_col2": "1,234,000",          // 見積要求額
        "header_col3": "1,200,000",          // 1次査定
        "header_col4": "...", "header_col5": "...",
        "header_col6": "...", "header_col7": "...",  // 〜最終査定
        "header_col8": "1,000,000",          // 前年度額
        "header_col9": "200,000",            // 比較金額(最終−前年度)
        "header_col10": "",                  // 比較率（空）
        "header_col11": "",                  // 予備（空）
        "items": [ /* 下位階層（施策・施策内訳） */ ]
      }
    ]
  }
}
```

### 列対応表（11 列）

| 列 | キー | 見出し | 値 |
|---|---|---|---|
| 1 | header_col1 | 事業/施策/施策内訳 | コード+名称（インデント） |
| 2 | header_col2 | 見積要求額 | `req` |
| 3 | header_col3 | 1次査定 | `a1` |
| 4 | header_col4 | 2次査定 | `a2` |
| 5 | header_col5 | 3次査定 | `a3` |
| 6 | header_col6 | 4次査定 | `a4` |
| 7 | header_col7 | 最終査定 | `af` |
| 8 | header_col8 | 前年度の額 | `prev` |
| 9 | header_col9 | 比較金額 | `af - prev` |
| 10 | header_col10 | 比較率 | （空） |
| 11 | header_col11 | 予備 | （空） |

> **他帳票との差異:** 共通ヘルパ `_transform_lines` は `fiscal_year` を和暦変換（`to_wareki`）し `report_date` を生成するが、**JFP1080 はこれを使わず独自実装**しており、`fiscal_year` は西暦の文字列、`report_date` は空文字。元号表記・帳票発行日が必要かは仕様確定時に要確認。

## 7. 制約・未確定事項（仕様確定時に要確認）

1. **col10 比較率が未算出**（空文字固定）。算出要否と式（最終査定/前年度 等）を確認。
2. **col11 予備の用途**が未定義（空文字固定）。
3. **`fiscal_year` が西暦のまま**（和暦変換なし）、**`report_date` が空**。元号表記・発行日の要否を確認。
4. **査定額カスケードで 0 を「未設定」扱い**するため、意図的な 0 円査定が前段値で上書きされる可能性。
5. スキーマの **`level` 未使用**（インデントは再帰深さで決定）。入力 `level` との整合方針を確認。
6. config の `item_fields`（header_col1〜11 の既定値定義）は `transform` 本体では未参照（行は手組み）。
