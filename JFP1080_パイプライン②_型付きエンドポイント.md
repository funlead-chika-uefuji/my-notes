# JFP1080 出力パイプライン② — 型付きエンドポイント

**作成日:** 2026-06-24
**対象帳票:** JFP1080 事業別予算査定一覧表（当初）
**プロジェクト:** `/mnt/c/claude/svg-editor/`

---

## 1. Python ファイル名と処理概要

**ファイル名:** `server/app/api/v1/external/print_external.py`

**処理概要:**
帳票 `code_id` ごとに **型付きの外部 API エンドポイント** を公開するルーティング層。「予算編成側から印刷へ何を投げるか」を OpenAPI（Swagger）で一意に明示することが主目的。URL を `code_id` と完全一致させ（1 code_id = 1 エンドポイント）、内部処理は既存の汎用 `POST /api/v1/external/print`（③ `create_print_job`）へ委譲する **薄い層**。JFP1080 は `post_jfp1080` ハンドラが受け口となる。

---

## 2. ファイル内の処理項目一覧

| 処理プロジェクト名（定義名） | 処理概要 | 処理内容 |
|------|---------|---------|
| モジュール docstring / 設計方針 | 層の位置付け宣言 | 「URL=code_id 完全一致」「汎用 print への委譲」「帳票追加は本ファイルへハンドラ1本追加（5〜10行）」の方針を明記 |
| import 群 | 依存の取り込み | `create_print_job`（③）、`PrintRequest` スキーマ、`get_api_key`/`get_db` 依存、`print_external_payload` から各 code_id 定数（`JFP1080_CODE_ID` 等）と各 Request クラス（`ProjectAssessmentInitialPrintRequest` 等）を取り込む |
| `router` | ルーター生成 | `APIRouter(tags=["印刷サブシステム連携"])` を生成 |
| `_PDF_RESPONSES` | API レスポンス定義 | Swagger 用に 200=PDF バイナリ / 202=ジョブ受付 / 404=テンプレート未検出 / 500=PDF 生成失敗 を宣言 |
| `_new_payload(req)` | 新形式ペイロード整形 | `req.model_dump(exclude={"sync"})` で `report_type + data + context` 形の dict を生成（JFP1080 含む新形式40帳票が使用） |
| `_to_response(code_id, form_data, sync, db, api_key)` | 汎用処理への委譲 | `PrintRequest(code_id, form_data, sync)` を組み立て、`create_print_job(...)`（③）を呼んで `Response` を返す |
| `post_jfp1080(request, db, api_key)` ★主経路 | JFP1080 エンドポイント | `@router.post(f"/{JFP1080_CODE_ID}", ...)` で公開。`ProjectAssessmentInitialPrintRequest` を受け、`_to_response(JFP1080_CODE_ID, _new_payload(request), request.sync, db, api_key)` を返す |
| `_legacy_payload(req)`（参考） | 旧形式ペイロード整形 | BRP9136/9146（資金計画書）専用。`fixed` + `lines` 形に整形（JFP1080 では不使用） |

---

## 3. 補足

- 本層自体に業務ロジックは無く、**型検証済みリクエストを汎用処理へ橋渡しするだけ**。実際のデータ準備・PDF 生成は③以降が担う。
- `post_jfp1080` が受け取る型（`ProjectAssessmentInitialPrintRequest`）の中身は **パイプライン①** で定義されている。
